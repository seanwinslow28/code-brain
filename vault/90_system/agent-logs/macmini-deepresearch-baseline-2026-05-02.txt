Phase 2 execution delta (2026-05-02 13:24)
------------------------------------------
The plan §3 Phase 2 Modelfile spec used `SYSTEM """/no_think"""`. On Ollama
0.22.1 this does NOT disable Qwen3's thinking mode. Ollama models thinking as
a separate top-level `thinking` field on the chat response, not as inline
`<think>` tags. The qwen3:14b stock template injects `/no_think` ONLY when
the API request includes `"think": false` — generic Ollama clients (LDR
included) don't pass that field, so the model defaults to thinking ON.

First two test calls measured: 502 then 330 tokens generated for a "say hi
in 5 words" prompt, all routed to a `thinking` field hidden from the user.
Wall time ~21s and ~14s respectively at 24 tok/s. For a 4-call research
synthesis run that's ~80s of pure thinking overhead per query.

Fix: patched the Modelfile to (a) unconditionally append `/no_think` at the
end of the last user message, and (b) pre-fill an empty `<think></think>`
block in the assistant prefix. This makes the model behave as non-thinking
by default for ANY caller, regardless of whether they pass the `think` API
field. `PARAMETER think false` is not supported in Ollama 0.22.1 (returns
"unknown parameter 'think'").

Post-patch verification:
  * "Say hi in 5 words" → "Hello, how are you today?" — 8 tokens, 282 ms,
    28.4 tok/s, thinking field empty.
  * "What is the capital of France?" → "The capital of France is Paris." —
    8 tokens, no thinking.

Tok/s 28.4 is within plan §3 Phase 2 expectation (25-35 on M4 base, faster
on M4 Pro). The Modelfile lives at:
  ~/Code-Brain/local-deep-research-stack/qwen3-14b-research.Modelfile

Add this finding as an execution delta to the macmini migration plan during
Phase 10 doc updates (mirroring the v3.21.0 §8 "thinking mode at LM Studio"
delta in you-are-a-senior-modular-pelican.md).

Phase 3 execution deltas (2026-05-02 13:30-13:39)
-------------------------------------------------
1. SETTINGS KEY SCHEMA SHIFT (LDR v1.5.6).
   Plan §3 Phase 3 step 8 listed `search.searxng_url` and
   `search.max_results_per_query`. Both return HTTP 404 in v1.5.6 — the
   per-engine schema moved them under
   `search.engine.web.searxng.default_params.{instance_url,max_results}`.
   Remaining keys (`llm.provider`, `llm.ollama.url`, `llm.model`,
   `search.tool`, `search.iterations`) keep the plan's names.

2. LLM.PROVIDER VALUE CASE.
   The /settings/api/llm.provider options list uses lowercase "ollama"
   even though LDR ships with the value defaulting to uppercase "OLLAMA".
   Setting to "ollama" is accepted (HTTP 200) and matches the canonical
   options-list value.

3. LDR EXACT-MATCH MODEL LOOKUP.
   Setting `llm.model = "qwen3-14b-research"` produces a research-start
   failure at t≈4s with metadata.error =
   "Model 'qwen3-14b-research' not found in Ollama". Ollama's /api/tags
   reports the model as `qwen3-14b-research:latest` (the `:latest` suffix
   is auto-appended on `ollama create`); LDR does an exact-string lookup,
   so the bare name doesn't match. Setting `llm.model =
   "qwen3-14b-research:latest"` succeeds. Recorded in configure_ldr.py
   as the canonical value.

4. AUTH RATE LIMITER (already documented in v3.21.0 §8).
   Multiple login attempts during register-debug → HTTP 429 sliding window.
   Cleared by `pkill -f ldr-web` + restart. Subsequent flow used a single
   login + cookie reuse for all calls. Recorded as confirmed gotcha.

5. CHECKBOX VALUE FORMAT.
   The /auth/register form has `<input name="acknowledge" value="true">`.
   `acknowledge=y` and `acknowledge=on` both yield HTTP 400 with "You must
   acknowledge that password recovery is not possible". Only the literal
   string `acknowledge=true` is accepted (matches the input's value attr).
