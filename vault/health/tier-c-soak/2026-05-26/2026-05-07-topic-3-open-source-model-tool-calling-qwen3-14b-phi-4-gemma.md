This research report evaluates the reliability of tool-calling capabilities in 14B-class open-source models—specifically Qwen3-14B, Phi-4, and Gemma 4—running on local Mac hardware via Ollama and LM Studio. The central thesis is that for headless tool-calling applications, the most reliable configuration is the Phi-4 model paired with an In-Process MCP (Model Context Protocol) pattern, as this combination minimizes latency and ensures seamless interaction with external systems.

The report details the performance and integration patterns for each model. Phi-4 and Gemma 4 are identified as highly reliable on Mac Mini and MacBook Pro hardware, both supporting the MCP protocol and OpenAI-compatible APIs through the standard tools parameter. For Gemma 4, community recipes suggest using an MCP server to translate between the model's function-calling format and the MCP tool discovery protocol. In contrast, Qwen3-14B presents significant challenges due to a custom command-style output format that many frameworks fail to recognize, often requiring custom configurations or tool definitions. While Qwen3-Coder has seen fixes for tool calling in environments like llama.cpp and Ollama, the standard Qwen3-14B remains inconsistent on Mac hardware. Community best practices for Qwen models include using Qwen-Agent and MCP configuration files to manage these complexities.

The research concludes with a clear recommendation for developers seeking stability: utilize Phi-4 with an In-Process MCP pattern to run the MCP server within the same process as the model. While Qwen3-14B remains a strong candidate for agentic applications due to its "thinking" and "non-thinking" modes, it is not recommended for headless tool-calling unless its formatting issues are resolved. The report leaves open the potential for Qwen3-14B to become a viable alternative if tool-calling fixes similar to those implemented for Qwen3-Coder are applied universally.
