The research report explores the functional distinctions between Ollama Modelfile SYSTEM prompts and runtime system messages when utilizing the Qwen3 model. The central thesis is that while both methods serve to define model behavior, they differ fundamentally in their persistence, flexibility, and implementation: Modelfile prompts act as a static, "baked-in" foundation for a model's persona, whereas runtime messages provide a dynamic mechanism to override or supplement that foundation during specific API requests or chat sessions.

The report supports this distinction by categorizing the two methods across several dimensions. Modelfile SYSTEM prompts are described as static and persistent, making them ideal for creating custom models with fixed personas, such as instructing a model to always follow a specific output format or strict guidelines. In contrast, runtime system messages are characterized by their flexibility, allowing for per-request adjustments to tailor responses to specific contexts. Implementation-wise, Modelfile prompts are defined using the `SYSTEM` keyword during model initialization, while runtime messages are passed via the API request payload. However, the report notes that runtime messages require more complex management to ensure consistency, and Modelfile prompts cannot be modified without recreating the model.

The report concludes by highlighting several unresolved contradictions and areas for further investigation. Specifically, there is conflicting information regarding whether the Qwen3-vl model supports Modelfile SYSTEM prompts, and there is ambiguity concerning whether prompt-based system messages should be viewed as an alternative or a preferred method compared to Modelfiles. Additionally, the report identifies a critical failure in RAG (Retrieval-Augmented Generation) tasks where Qwen3 models reportedly fail to process tasks when using standard system prompts. The researchers recommend clarifying support for Qwen3-vl, resolving the ambiguity between prompt and Modelfile methods, and investigating the reported RAG task failures.
