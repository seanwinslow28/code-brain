This report details the technical configuration required to integrate the Pi terminal-based coding agent (pi.dev) with a local or LAN-reachable Ollama instance. The central thesis is that Pi treats Ollama as a first-class, OpenAI-compatible provider, allowing users to leverage local hardware—such as a Mac Mini or Alienware desktop—to run agentic workloads with low latency and reduced "agentic hover."

The integration relies on configuring the `~/.pi/agent/models.json` file with an OpenAI-compatible endpoint shape, specifically using the `/v1` path (e.g., `http://localhost:11434/v1`). Key implementation details include using `OLLAMA_HOST=0.0.0.0` to enable LAN access via hostnames like `mac-mini.local` or Tailscale MagicDNS addresses. For performance and reliability, the report recommends using DeepSeek-Coder-V3 for stable tool-calling, or Qwen3-14B for fast tasks, and suggests managing the context window via the `num_ctx` parameter in an Ollama `Modelfile` rather than the Pi config. Additionally, the report highlights that local M4 Pro hardware can achieve 40–70 tokens per second with a time-to-first-token of under 200ms, outperforming cloud providers in latency.

The report identifies several "gotchas" and areas for further investigation, such as the need to set a `keep_alive` environment variable to prevent models from unloading during rapid edits and the importance of ensuring the base URL includes `/v1` to maintain streaming compatibility. While Llama-4 is noted as an emerging performer, the report observes that documentation regarding its tool-call stability with specific Pi "skills" is still evolving. Finally, the author poses an open question regarding potential latency issues when routing requests specifically from an Alienware desktop to a Mac Mini.
