## Going Forward — Where Completion Writes Land

When the parent roadmap's task scaffolding makes it ambiguous which file to write into, default to writing **here**, then add or update the stub in the parent.

**Three flavors of completion write, in priority order:**

1. **A whole task closes** (every step ✅, verification gate satisfied) → move the full task body to `## Completed Tasks` above; replace the parent's task block with a 2–3 line stub linking back. This is the cleanest case; treat Task 0 above as the canonical example of the resulting stub-and-record shape.

2. **A multi-day Workstream wraps inside a still-active task** (e.g., Task 8 Workstreams A/B/C done but B7 gate pending; Task 11 deploy done but Substack post 2 pending) → add a new H3 under `## Amendments Log` here capturing the ship narrative; leave the active task body in the parent intact. The parent stays focused on what's *still* in flight; this file holds the *what shipped* narrative.

3. **A discrete Cowork session amendment** (one-session scope changes, decision locks, expansions, etc. — the kind of entry that would previously have been added to the parent's frontmatter `amendments:` YAML list) → add a new H3 under `## Amendments Log` here. The parent's frontmatter no longer carries the amendments list; this file is the canonical home.

**Anchor convention for entries under `## Amendments Log`:** `### YYYY-MM-DD [(qualifier)] — short topic`. Qualifier disambiguates same-day entries (`(evening)`, `(later)`, `(final)`, etc.). Topic captures the load-bearing change in ≤8 words. This keeps anchors stable and predictable when the parent roadmap references back.

**Wikilinks and relative paths:** This file lives in the same directory as the parent (`vault/20_projects/prj-job-hunt-2026/onwards-and-upwards-5-4-26/job-hunt-2026-roadmap/`). Relative paths resolve identically from both files — `../../target-companies.md` works from here exactly as from the parent. No path rewriting needed when content moves between files.
