# Resume Design Notes

The companion `.md` is the source of truth for content. The `.docx` is the deliverable Sean edits and sends. This file is the **why** behind the layout.

## The brief

Sean asked for "simplicity with a splash of color for some personality." Target audience: AI-forward companies hiring senior PMs. Research synthesis is in chat history (May 2026 session); the conclusions are codified here.

## Design system — locked-in choices

| Choice | Value | Why |
|---|---|---|
| Layout | Single column, no sidebar | Jobscan 2025 parse audit: single-column = 93% field completeness, multi-column = 86%. Canva-style sidebars score as low as 18% on Workday. Two-column is dead for ATS in 2026. |
| Length | Two pages | Correct for 10+ years experience. One-page is for <10 years. |
| Accent color | Navy `#0C447C` | "Splash of color" 2026 convention = one accent in 2–3 places. Navy reads confident, finance/tech-native, and is the default for senior PM resumes targeting AI-forward companies. |
| Where accent appears | Name, 5 section headers, single hairline rule under contact block, skill-category labels | Anything more reads as "designed in Canva." The hairline rule + uppercase tracked section headers are the entire design language. |
| Font | Calibri throughout | Universally available on Word installs (Mac + Windows), renders identically across platforms, ATS-safe. Inter / Aptos are equally defensible — Find & Replace will swap cleanly. Avoid serif (reads legal/academic, not AI-forward). |
| Font sizes | Name 26pt / section headers 10.5pt uppercase / subheads 10.5pt bold / body 10pt / dates+location 9.5pt muted | Jobscan: never go below 10pt. 10pt body is the minimum that still scans cleanly. |
| Margins | 0.55" all sides | Standard is 1". Tightened to 0.55" to fit two-page target without trimming content. Anything below 0.5" starts triggering ATS edge-clipping risk. |
| Section order | Summary → Selected AI Artifacts → Work Experience → Leadership → Education → Skills | **Artifacts before Work Experience is deliberate.** Explicit Teal / Resume Worded guidance: when public artifacts (npm package, OSS toolkit) are stronger signal than the last job title, they belong in the first third of page 1 where AI hiring managers scan for "this person actually builds with AI" proof. |
| Bullets | LevelFormat.BULLET via numbering config | Never use unicode `•` characters inline — breaks ATS. |
| Dates / location | Right-tab-stop aligned at 10080 DXA | Position depends on content width. If margins change, this number must update: `content_width_DXA = 12240 − (left_margin + right_margin)`. |

## What we explicitly rejected

These were considered and discarded — don't reintroduce them without revisiting the evidence:

- **Skill bars / proficiency percentages / star ratings.** ATS-invisible, recruiters read them as fluff.
- **Icons next to bullets or contact items.** ATS can't parse them; they signal "Canva resume."
- **Photo / headshot.** US market — bias risk and ATS breakage.
- **Two-column sidebar.** See parse-rate evidence above.
- **Multiple accent colors / rainbow.** Single accent only.
- **Serif body font (Garamond, Charter, Source Serif).** Reads legal/academic, not AI-forward.
- **One-page condensed.** Would have required cutting ~30% of bullets including the New York Life detail. Defended two pages.
- **Moving Selected Artifacts after Work Experience.** Safer for traditional recruiters but weaker positioning for AI-forward roles. Sean prioritized signal over safety.

## Editing notes for future Claude sessions

The `.docx` is generated by `build_resume.js` (lives in the outputs scratch folder during build; not committed). The current `.docx` was built with these parameters — if Sean asks to regenerate or edit:

1. **Editing in Word directly is fine for content.** Bullets, dates, job titles, and bullet content can be edited in Word without breaking the design. The list-numbering reference is single and stable.
2. **Editing the design requires the build script.** Section headers are styled as direct text runs (not Heading 1/2 styles) because docx-js heading styles don't accept custom border-bottom properties cleanly. To change the section-header look, edit `sectionHeader()` in the build script and rebuild.
3. **Font swap (Calibri → Aptos / Inter):** Find & Replace "Calibri" → target font across the doc. No other changes needed.
4. **Margin changes:** If margins change, update the tab-stop position in `titleWithDate()` and `roleLine()` to `12240 − (left + right)` in DXA. Otherwise dates wrap to a second line.
5. **Adding a new section:** Use `sectionHeader("Name")` followed by content. Section headers are ALL CAPS with letter-spacing 30 (character spacing in docx units).
6. **The `_DESIGN_NOTES.md` (this file) is the durable record.** The build script may be regenerated/lost across sessions; this file should not be deleted.

## ATS sanity checklist

Before sending the resume to a real ATS pipeline, re-verify:

- [ ] Open the `.docx` in Word, save as `.txt`, and confirm all content survives in linear reading order.
- [ ] Run through a free Jobscan / Resume Worded ATS check at least once per job-application batch.
- [ ] Confirm no images, no text boxes, no tables-used-as-layout. (Current design has none of these.)
- [ ] Confirm contact info is in body text, not in a header/footer. (Current design: contact line is body, not header — correct.)

## What "good" looks like

If a future session is asked to redesign the resume, the test is: would this design pass for a Stripe, Linear, or Notion in-house PM resume? If it doesn't read native to that visual lineage — single column, sans-serif, one accent color, no icons, no graphics, heavy white space, dense quantified bullets — it's wrong for Sean's target audience.
