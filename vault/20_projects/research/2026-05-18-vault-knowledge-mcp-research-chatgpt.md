## Embedding strategy verdict

Pick **JS-native inside the MCP server**, not a Python sidecar.

This is the cleaner fit for both the build constraints and the recruiter experience. Your existing published MCP is TypeScript, your intended stack is TypeScript + Node 22 + MCP SDK 1.29.0, and the best local-first reference implementations on the official side are already `npx`-first for TypeScript servers. AnthropicŌĆÖs official local-server docs and official server repo both normalize subprocess-based local installation for Node packages. The SQLite `npx` adapter exists for exactly the same reason: avoiding an extra Python runtime in environments where `uvx` is absent or inconvenient. ŅłĆciteŅłéturn8search0Ņłéturn7search12Ņłéturn5view1Ņłéturn34view1Ņłü

The case against a Python sidecar is not that Python MCPs are bad. `mcp-obsidian` is clearly successful, and the official `fetch` server is Python-based. The issue is cloning friction for a recruiter or hiring manager. A JS-native build keeps the stack to Node plus Claude Desktop config plus a demo vault. A Python sidecar usually means one more runtime, one more package manager (`uv`/`uvx` or `pip`), and more setup ambiguity around env files or API plugins. That is acceptable when the Python dependency is the product; it is a distraction when the product is the reasoning surface. ŅłĆciteŅłéturn33view0Ņłéturn25view1Ņłéturn5view1Ņłü

In practical terms, the recruiter-clone path should be: `npm ci`, `npm run build`, and one pasted config block pointing to either `node dist/index.js` or `npx -y @swins/vault-knowledge-mcp`, plus a checked-in synthetic demo vault. That is a materially better story than ŌĆ£install Node, install Python, install uv, configure REST plugin, and then run a bridge.ŌĆØ ŅłĆciteŅłéturn24view0Ņłéturn34view1Ņłéturn33view0Ņłü

## State management verdict

Use the **existing read-only `.vault-index.db` and edge tables** for v0. Do **not** build a new embedding index inside the server.

This is the strongest architectural fit with your current assets and the clearest way to make the artifact feel like a customer-facing surface for the work you already did in Phase D. The most relevant thin-server references do not duplicate state; they expose an existing substrate. `server-postgres` exposes an existing database with read-only queries and schema resources. `mcp-server-sqlite-npx` exists to expose an existing SQLite file. `server-filesystem` exposes existing files under a bounded path. ## Privacy boundary pattern

Use an **allowlist pattern**, enforced server-side, with explicit `VAULT_ROOT` and `DB_PATH` inputs. Do not rely on a blocklist. Do not rely on MCP Roots alone.

This is the dominant safe pattern in the server set you actually want to copy. `server-filesystem` explicitly restricts all operations to allowed directories and recommends Roots for dynamic updates, but still treats server-side access control as real enforcement. The MCP Roots spec also says roots are informational guidance rather than a hard access-control mechanism, which means you should use them as a hint, not as your only barrier. `mcp-server-sqlite-npx` similarly requires an explicit database path in the launch config. `eddmann/obsidian-mcp` requires an explicit vault repo/branch and env configuration. These are all allowlist-shaped boundaries. ŅłĆciteŅłéturn24view0Ņłéturn7search1Ņłéturn7search10Ņłéturn34view1Ņłéturn33view1Ņłü

For `vault-knowledge-mcp`, the clean pattern is: fail closed unless both a vault root and a database path are explicitly supplied; normalize and canonicalize paths on startup; verify the SQLite file lives inside the approved root or a small set of approved roots; and expose only read-only operations. The server may optionally ask for client roots and intersect them with the configured allowlist, but it should never grant access based on roots alone. ŅłĆciteŅłéturn24view0Ņłéturn7search1Ņłéturn7search10Ņłü

This is not just a security recommendation. It is also a positioning recommendation. A read-only, fail-closed allowlist makes the README and Loom much stronger because it communicates judgment: you understand that local MCPs are powerful, and you intentionally narrowed the blast radius. AnthropicŌĆÖs security docs repeatedly push least privilege, and community discussion around stdio servers has only made that more salient. ŅłĆciteŅłéturn6search0Ņłéturn6search6Ņłéturn6search2Ņłü

## Tool count verdict

Keep the public surface at **three core capabilities**, but do **not** spend all three on tools.

The survey distribution is bimodal. Thin adapters like `server-fetch` and `server-postgres` sit at one tool. The SQLite adapter is effectively a tiny two-tool pattern. Workflow products like the Obsidian servers jump to 17ŌĆō18 tools. `server-memory` lands at nine because it is a full graph-mutation surface. There is no compelling evidence in this set that six tools is the ŌĆ£rightŌĆØ middle ground for a one-week portfolio server. ŅłĆciteŅłéturn25view1Ņłéturn25view2Ņłéturn34view1Ņłéturn35search8Ņłéturn33view1Ņłéturn33view0Ņłü

What I would change is not the count, but the *type allocation*. `get_article(slug)` is better modeled as a **resource** than as a tool. The MCP docs distinguish resources as shareable contextual data and tools as executable actions; `server-postgres` follows that split by exposing table schemas as resources and the query function as a tool. If you keep article retrieval as a tool, you are spending one-third of your public surface on something that feels generic. If you move article retrieval to a resource namespace like `vault://article/<slug>`, you free a tool slot for the more portfolio-relevant action: `explain_contradiction` or `trace_edge_path`. ŅłĆciteŅłéturn7search5Ņłéturn25view2Ņłü

So the verdict is: **three public capabilities, but likely two tools plus resources for the retrieval layer**. If you insist on a tool-only v0 because you want parity with your previous MCP, then keep it to three tools, not six. But the more idiomatic and more differentiated version is to make retrieval a resource and reserve tools for reasoning work. ŅłĆciteŅłéturn7search5Ņłéturn7search19Ņłéturn25view2Ņłü

**Move `get_article(slug)` out of the tool surface and expose articles as MCP resources; use the freed slot for `explain_contradiction` or `trace_edge_path`.**
