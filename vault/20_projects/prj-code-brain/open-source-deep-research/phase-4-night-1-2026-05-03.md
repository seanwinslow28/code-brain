## Tech-debt items (observed during smoke test, log for v3.26.0)

These are NOT blocking. None of them affect correctness. They're polish items.

### TD1 — Slug derivation produces ugly filenames and digest links (P2)

**Symptom:** The smoke test wrote to `vault/20_projects/research/2026-05-03-you-are-a-senior-research-analyst-at-the-intersection-of-ai.md`. The slug is the first 60 chars of the prompt, which begins with the role prefix ("You are a senior research analyst..."). Same issue surfaces in the daily-note digest line — the wikilink anchor text repeats the entire role prefix.

**Root cause:** `agents-sdk/scripts/gemini_dr.py:slugify()` takes `query[:60]` without stripping XML tags or role prefixes. Works fine for terse queue lines like "What are the practical differences between A and B?" — fails on prompt-engineered queries.

**Fix options:**
- (a) Add `--slug` CLI flag override; skill computes a clean slug from the user's intent before passing to helper.
- (b) Detect XML structure in the query and slugify the `<your_task>` content (or first non-tag line) instead of the role prefix.
- (c) Have the helper extract the topic from the prompt's first imperative sentence (heuristic).

(a) is the simplest. Recommend that for v3.26.0.

### TD2 — `datetime.utcnow()` deprecation warning in helper (P3)

**Symptom:** `gemini_dr.py:565: DeprecationWarning: datetime.datetime.utcnow() is deprecated and scheduled for removal in a future version.`

**Fix:** Replace `datetime.utcnow().isoformat() + "Z"` with `datetime.now(datetime.UTC).isoformat()` (the timezone-aware form). One-line change. Same call site that writes the ledger entry's `created` field.

### TD3 — Frontmatter `question:` field includes the entire prompt (P3)

**Symptom:** When a prompt-engineered query is fired (1000+ tokens of XML-tagged structured input), the entire prompt lands in the report's frontmatter `question:` field. The frontmatter alone becomes ~5KB and visually buries the body when opened in Obsidian.

**Fix options:**
- (a) Truncate `question:` to first 500 chars + ellipsis; preserve the full prompt in a separate `<details>` block in the body.
- (b) Move the full prompt to a `prompt:` field at the bottom of the frontmatter (Obsidian's frontmatter renderer collapses long fields better when they're below the line of human-relevant fields like `date`, `tier`, `cost_usd`).

(b) is less invasive. v3.26.0 polish.

### TD5 — CRITICAL bug DISCOVERED + FIXED in this session: helper grabbed wrong outputs index, lost answer body (P0, fixed in commit below)

**Symptom:** Smoke test report at [vault/20_projects/research/2026-05-03-you-are-a-senior-research-analyst-at-the-intersection-of-ai.md](vault/20_projects/research/2026-05-03-you-are-a-senior-research-analyst-at-the-intersection-of-ai.md) appeared to contain only the prompt + sources, with no actual research answer. Sean caught it during session wrap-up.

**Root cause:** Gemini Deep Research returns 3 outputs for a `completed` interaction:
- `outputs[0]` = `ThoughtContent` (Gemini's internal reasoning, no `.text` attribute)
- `outputs[1]` = `TextContent` — the actual research answer body (~22,290 chars in our case)
- `outputs[2]` = `TextContent` — the sources block (~8,130 chars)

`scripts/gemini_dr.py:poll_interaction` (line 364) used `outputs[-1].text` which grabbed only the sources block. The 22,290-char answer body was silently discarded. Phase 1 spec said "capture interaction.outputs[-1].text" — that was wrong about the API shape. The Phase 1 spec reviewer noted the empty-outputs edge case but missed this multi-output case.

**Why tests didn't catch it:** `test_polling_happy_path` mocked `outputs = [completed_text]` (a single-element list) — so `outputs[-1]` returned the right thing in tests but the wrong thing in production.

**Fix applied THIS session (no extra cost):**
1. Re-fetched the smoke test interaction from Gemini's storage (Gemini retains results for some period; `client.interactions.get(id)` returned the same outputs as the original poll). Concatenated all `.text`-bearing outputs and rewrote the report file. File grew from 19,886 → 42,201 bytes. The actual answer is now in the file.
2. Patched `scripts/gemini_dr.py:poll_interaction` to iterate ALL outputs and concatenate text-bearing ones, joining with `\n\n`.
3. Added regression test `test_polling_concatenates_multiple_text_outputs` that mocks the [ThoughtContent, TextContent, TextContent] shape and asserts both answer + sources are returned. Test count: 32 → 33.

**Impact assessment:** Topic 2 (when fired tomorrow) and topic 4 (DR Max, when fired night 3) WILL benefit from this fix. Smoke test was the ONLY pre-fix run; it has been recovered. No data loss.

**Lesson for Phase 1 reviewers:** When mocking SDK objects, vary the SHAPE of the mocked structure (single vs multi-element arrays, different content-type subclasses) — not just the values. The single-element happy-path test gave false confidence.

## Future enhancement: load `prompt-engineering` skill into `deep_researcher`'s system prompt

Sean's idea (from this session's conversation): the `deep_researcher.py` agent currently passes the queue line verbatim to LDR. Same for `gemini_researcher.py` → `gemini_dr.run()`. If the queue line is terse, output quality suffers. Two improvement paths:

**Option A — Pre-process the queue line through Claude before firing.** Add a "prompt rewriter" step that takes the queue line + the `prompt-engineering` skill body and produces an engineered version. Then fire that. Cost: 1 extra Sonnet call per queue item ($0.05–0.20). Benefit: consistent prompt quality without manual engineering.

**Option B — Embed prompt-engineering principles in the `deep_researcher` agent's own system prompt.** When the agent runs, it loads the `prompt-engineering` skill body as part of its system context, then reads the queue line, internally engineers it, and fires. Same outcome as A but in one agent invocation.

**Recommendation:** A is cleaner architecturally (separation of concerns: rewriter agent, then research agent). B is cheaper. For now, manual engineering for cloud calls (DR / DR Max) is the best ROI — cloud spend is the high-stakes path. Keep LDR queue lines simple; LDR's iteration handles less-engineered queries reasonably well.

Future Sean: revisit after 2-3 weeks of usage data. If many LDR runs produce thin output, build A as a v3.27.0 enhancement.
