## What this is

Critiques from two external reasoners (gpt-5.5 via Codex CLI, Gemini 3 via Anti-Gravity CLI) of [[portfolio-projects-for-interview-prep]]. The synthesizer describes what the concept is; this expansion proposes what's missing.

## From Codex (gpt-5.5)

1. **Add “proof-of-work taxonomy” anchored on Patrick McKenzie’s essay “Don’t Call Yourself A Programmer”**

   The concept says “portfolio projects,” but it does not yet classify *what kind of hiring evidence* each artifact produces. Add a taxonomy: `demo artifact`, `operating artifact`, `taste artifact`, `judgment artifact`, `systems artifact`.

   Exemplar: Patrick McKenzie, **“Don’t Call Yourself A Programmer”**. The relevant move is not the career advice; it is his insistence that candidates be legible through business value, not self-description.

   This unlocks a **portfolio one-pager / interview packet** where each project answers: “What decision does this help a hiring manager make about me?” Current concept risks listing impressive builds. This would force each artifact into a hiring-manager inference: “Sean can operate autonomous agent systems,” “Sean can ship AI-native product loops,” “Sean can debug reliability under constraints.”

2. **Add “case-study spine” anchored on Richard Feynman’s “Cargo Cult Science”**

   The concept needs an anti-demo criterion: what would make a portfolio artifact *look* impressive while proving nothing? Add a `cargo-cult portfolio` warning mode: demos that hide failure modes, omit baselines, lack before/after evidence, or cannot be reproduced.

   Exemplar: Richard Feynman, **“Cargo Cult Science”**. The specific missing pattern is “show the thing that could have disproved you.” For Sean, that means every flagship artifact should include: baseline, intervention, failure case, evaluation, and what changed after critique.

   This unlocks **interview-ready technical case studies** instead of showcase pages. For example, “Judge Layer Retrofit” should not just say `79 tests passing`; it should show the prior failure mode, the eval harness, examples the judge caught, examples it missed, and the product judgment behind the threshold. That produces senior IC signal, not just builder signal.

3. **Add “portfolio as narrative wedge” anchored on Kathy Sierra’s “Badass: Making Users Awesome”**

   The concept frames projects as proof of Sean’s skills, but misses the sharper question: *who becomes more capable because this exists?* Add a “user-transformation” layer to every artifact: before-state, new capability, repeated workflow, and why the artifact changes someone’s behavior.

   Exemplar: Kathy Sierra, **“Badass: Making Users Awesome.”** The transferable idea is that the product is not the hero; the user’s upgraded capability is. For Sean’s portfolio, the hiring manager should not see “Sean built an agent dashboard.” They should see “a solo operator can run a low-cost autonomous knowledge fleet with observable failure modes.”

   This unlocks **Substack essays and product narratives** that avoid generic “look what I built” writing. Sentence pattern: “Before this, the user had to ___; after this, they can ___; the system earns trust by ___.” That would let Sean turn Code-Brain, 16BitFit, and intent-engineering into role-targeted artifacts for AI-PM, agentic-engineering IC, and senior product roles without sounding like a tools résumé.
