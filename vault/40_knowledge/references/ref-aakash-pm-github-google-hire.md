## 1\. What is Github and Why do Devs Use It

Github is like the Google Docs for software projects.

When a developer writes code, they save it to GitHub. When another developer wants to improve that code, they can see the full history of changes, suggest edits, and track what worked and what didn’t.

Just like Google Docs for PMs (except even better with history).

And just like every PM writes docs, but not all use Google Docs, every developer uses Git, but not every developer uses Github.

There are alternatives like GitLab and Bitbucket. But they all run on Git underneath.

![Demystifying Git vs GitHub vs Atlassian Bitbucket vs GitLab](https://substackcdn.com/image/fetch/$s_!jAZK!,w_1456,c_limit,f_webp,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F0d87c986-942f-480a-b088-cc249647f2f5_1280x640.jpeg)

Demystifying Git vs GitHub vs Atlassian Bitbucket vs GitLab

Git provides two things professional developers can’t work without:

1. **Version Control:** Every change is saved. If something breaks, developers can revert to any previous version. They never lose work.
2. **Collaboration:** Multiple people can work on the same codebase without overwriting each other’s changes. GitHub tracks who changed what and when.

Github adds a third benefit on top: it’s a **public portfolio.** Open source projects live on GitHub. Developers showcase their work, contribute to popular projects, and build reputation through their profile.

For developers, GitHub is standard infrastructure. For PMs, it’s optional. That’s exactly why it matters.

When you have a GitHub profile as a PM, you signal something hiring managers rarely see: **you build things**. You understand the tools your engineering team uses daily. You’ve crossed the line from talking about products to shipping them.

Most PMs list skills on their resume. PMs with GitHub profiles prove those skills with working code.

That proof is what opens doors.

## 3\. How to Build Your PM Github

You don’t need to match Shubham’s star count. You need projects that solve real problems and documentation that shows your thinking.

GitHub feels intimidating if you’ve never used it. But with AI coding tools like Cursor and Claude Code, the technical part is handled for you.

I created a starter kit repo to go with this post: [pm-github-starter-kit](https://github.com/aakashg/pm-github-starter-kit). It has profile and project README templates, Cursor/Claude Code prompts, and a 3-week checklist. Fork it and follow along.

### Day 1: Set Up Your Profile and Tools

**Step 1: Create your GitHub account.** Go to github.com and sign up. Use a professional username, ideally your actual name. Hiring managers will see this.

**Step 2: Fill out your profile.** Your bio should follow this structure:

> AI PM building \[specific thing\]. Shipping \[type of projects\].

Add your location and link to your LinkedIn.

**Step 3: Create a profile README.** Create a repository named exactly the same as your username. This becomes your profile’s landing page. The starter kit has a template you can fill in: *what you build, what problems you solve, what you’re working on now*.

**Step 4: Install Cursor or Claude Code.** For Cursor, download from cursor.sh. For Claude Code, follow the instructions at docs.claude.com/claude-code. Connect your GitHub account when prompted. Both tools will ask for GitHub permissions. Grant them. This lets the AI push code directly to your repos.

### Day 2: Your First Repository

Now you build something real. Start with a project from my AI agent guide or prototyping guide, or pick one from the 50 Project Ideas infographic below.

Here’s the workflow you’ll use every time: **Build, Commit, Push**.

**Build your project.** Open Cursor and press Cmd/Ctrl + K (or use Claude Code in your terminal). Tell the AI what you want:

```markup
Build a simple email drafting agent using OpenAI's API.
Include:
- A Python script that analyzes incoming emails
- A function that generates draft responses
- A README with setup instructions
```

The AI writes the code and creates the files. The starter kit includes prompt templates for this step.

**Commit your changes.** A commit is a save point. It captures what changed and why.

```markup
Commit these changes with message:
"Add initial email drafting agent with OpenAI integration"
```

The AI handles the git commands. When you say “commit and push,” it runs `git add .`, `git commit -m "Your message"`, and `git push origin main` for you.

**Push to GitHub.** This uploads your commits so others can see them.

```markup
Push these commits to GitHub
```

Done. Your code is live on your profile.

**Then keep building.** Each session creates a commit. Each commit shows up on your profile:

```markup
Session 1: Build
→ "Initial email agent with OpenAI integration"

Session 2: Add Features
→ "Add sentiment analysis feature"

Session 3: Documentation
→ "Add detailed README with demo and tradeoffs"
```

Two rules for clean commits. First, commit related changes together. “Add email sentiment analysis feature” with the feature file, updated requirements, and README update is one commit. Don’t mix random changes. Second, write clear messages. “Add meeting preparation agent” is good. “Updates” is bad.

### Week 2: Create Your Flagship Project

This is your awesome-llm-apps equivalent. Pick one pain point you experience as a PM and build a solution.

Types that work: PM automation tools (scripts that generate PRD sections, meeting summaries, competitive analysis), AI agent templates (ready-to-deploy agents for common PM tasks), evaluation frameworks (tools for testing AI outputs systematically), or integration examples (connecting AI tools to PM workflows like Notion, Jira, or Slack).

The goal: create something **10 other PMs** would actually use.

### Week 3: Documentation and Polish

This is where most PMs lose. They build something decent, then write a three-sentence README and call it done.

Your README is **your interview with the hiring manager**. Use the project README template from the starter kit. The key sections:

- **The Problem**. What pain point does this solve? Be specific about who has this problem.
- **The Solution**. What does the tool do? How does the user interact with it?
- **Tradeoffs and Decisions**. This section is what separates a PM GitHub from an engineer’s. Explain 1-2 decisions you made and why. What did you consider? What did you choose and why?
- **What I Learned**. Real insights. Not “ *I learned a lot about APIs*.” Something like: “ *Clustering user feedback by embedding similarity catches patterns keyword matching misses, but it groups unrelated complaints when the sample size is under 50.*”

Add screenshots, a demo GIF, or example outputs. Pin your best 2-3 repos. Add your GitHub link to LinkedIn and your resume.

### Handling Common Issues

These come up as you build. Cursor and Claude Code handle all of them:

- **“I made changes but don’t want to commit yet”**
	Tell the AI “Stash my current changes.” It saves them temporarily. Say “Apply my stashed changes” to get them back.
- **“I want to undo my last commit”**
	Tell the AI “Undo the last commit but keep the changes.” It reverts the commit but keeps your files.
- **“I want to experiment without breaking things”**
	Tell the AI “Create a new branch called experiment-with-claude.” Make your changes there. If it works, merge it back. If not, delete the branch.

You now know how to create repositories, build projects, commit changes, and push to GitHub. Everything else is variations on these basics.

## 4\. What to Avoid

I’ve reviewed hundreds of PM GitHubs at this point. Some through my cohort, some through hiring, some because people DM me asking for feedback.

Here are the patterns that hurt candidates.

![](https://substackcdn.com/image/fetch/$s_!UjP4!,w_1456,c_limit,f_webp,q_auto:good,fl_progressive:steep/https%3A%2F%2Fsubstack-post-media.s3.amazonaws.com%2Fpublic%2Fimages%2F7ff48d4b-acf5-4cf1-914d-ab60b82ae83c_1190x1144.png)

### Pattern 1 - The Copy-Paste GitHub

I can spot these in **10 seconds**. Every file looks like it came from the same ChatGPT session. The README describes the code in vague terms that could apply to any project. There’s no “Tradeoffs” section because the PM never made any tradeoffs. They just accepted whatever the AI generated.

Hiring managers catch this even faster than I do. The fix: after the AI generates code, change at least one design decision. Document why you changed it. If you can’t explain why your project uses OpenAI instead of Claude, or why you chose to cluster feedback by embedding similarity instead of keyword matching, you don’t understand *what you built*.

### Pattern 2 - The Joke Project

A chatbot that tells dad jokes. A “motivational quote generator.” An AI that writes haikus about your mood.

These prove you can prompt an LLM. They prove nothing about PM skills. Every project on your GitHub should solve a problem that a real person at a real company would pay to have solved. A feedback clustering tool that saves a PM 3 hours a week? That’s a project. A joke bot? That’s a weekend tutorial you should have kept *private*.

### Pattern 3 - The Green Square Farm

Some PMs learn that GitHub shows a contribution graph and decide to game it. They make 50 empty commits, push meaningless “update” messages, and hope the green squares impress someone.

The squares are **clickable**. Hiring managers click them. When they see “update README” forty times with no actual changes, that tells them more about you than an empty graph would have.

One real project per week beats a fully green graph with nothing behind it.

### Pattern 4 - The Engineer Cosplay

Your GitHub should not look like a software engineer’s. If you’re writing custom data structures, optimizing algorithms, or building microservices architecture, you’re proving the wrong thing.

Let the AI write clean code. You focus on three things: the problem statement (*why does this matter*?), the user experience (*how does someone actually use this*?), and the documentation (*what decisions did you make and why*?). A well-documented project with simple code beats a complex codebase with no README every time.

### Pattern 5 - The Private Repo Graveyard

Some PMs build useful internal tools at work and keep everything in private repos. Great for your job. Does nothing for *your* career.

If you’ve built something useful internally, rebuild a generalized version as a public repo. Strip out anything company-specific. Your internal meeting summarizer becomes a public “AI Meeting Notes” template. You get the portfolio piece without leaking anything sensitive.

### Pattern 6 - The One-and-Done

Shipped a repo 8 months ago. Haven’t touched GitHub since. Your contribution graph shows one green week in a sea of gray.

Hiring managers notice recency. A GitHub that’s been active in the last month signals you’re building with the current generation of AI tools. One that’s been dormant since last summer signals a weekend project you *abandoned*.

Build one small thing per week. Even a documentation update or a new prompt in your prompt library counts. **Consistency compounds**.
