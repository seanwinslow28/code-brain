# AI-Optimized Vault Architecture

## Purpose

Design Obsidian vault structures that serve both human browsing and AI retrieval. Define folder hierarchies, standardized YAML frontmatter schemas, atomic note principles, MOC linking strategies, and RAG-ready templates that maximize retrieval accuracy.

## When to Use

- Setting up a new Obsidian vault for AI integration
- Restructuring an existing vault (the "500 unstructured notes" case)
- Designing YAML frontmatter schemas for AI filtering
- Creating note templates optimized for embeddings
- Building Map of Content (MOC) index notes

| Scope | Convention | Example |

## YAML Frontmatter Schema

**CRITICAL: Flat YAML only.** No nested objects in frontmatter. This ensures compatibility with Dataview queries and simple grep-based parsing.

```yaml
---
type: project            # values: daily, weekly, project, meeting, reference, idea, concept, moc, dashboard
domain: [creative-studio] # values: claude-mastery, product-management, creative-studio, life-systems, design-team, vault
status: active           # values: backlog, draft, active, blocked, review, completed, archived
context: 16bitfit        # Short project/topic identifier for quick filtering
energy-level: medium     # values: low, medium, high (for task matching)
ai-context: "One-sentence summary for AI retrieval and Dataview display"
review-date: 2026-03-06  # Next review date
created: 2026-02-20
---
```

### Field Purposes

| Field | Purpose | AI Use |
|:------|:--------|:-------|
| `type` | Categorize note kind | Filter context by type |
| `domain` | Map to Claude Code workspace | Scope searches to domain |
| `status` | Track lifecycle stage | Find active/blocked items |
| `context` | Quick identifier | Match project notes to repos |
| `energy-level` | Task difficulty | Match tasks to energy |
| `ai-context` | Pre-computed summary | Low-cost retrieval without reading full note |
| `review-date` | Staleness detection | Surface notes needing attention |

| Type | Required Fields | Optional Fields |
|------|----------------|----------------|

## HTML Comment Anchors

All templates include HTML comment anchors for PATCH-style programmatic writes:

```markdown
<!-- anchor-name -->
```

Claude Code injects content **below** the anchor, leaving the comment in place for future writes. This avoids full-file rewrites.

Key anchors by template:
- **Daily:** `<!-- jira-log -->`, `<!-- claude-sessions -->`, `<!-- side-projects -->`
- **Weekly:** `<!-- auto-wins -->`, `<!-- auto-blockers -->`, `<!-- auto-decisions -->`
- **Project:** `<!-- status-update -->`, `<!-- git-commits -->`

## Atomic Note Principles for Embeddings

Atomic notes improve RAG retrieval precision. When a vector search retrieves a note, it gets only the relevant concept, not 10 pages of mixed topics.

1. **One idea per note.** "Cosine Similarity" not "Math Overview"
2. **Semantic integrity.** The note makes sense in isolation. Avoid orphaned pronouns in the opening sentence
3. **Descriptive titles.** "Optimization Techniques for Vector Embeddings" not "Optimization"
4. **Heading structure.** Use clear H1/H2 headers. AI models chunk by headers for retrieval
5. **Context in frontmatter.** Include `ai-context` for hybrid search (keyword + vector)

## MOC (Map of Content) Patterns

MOCs live in `vault/05_atlas/` and provide structured navigation per domain. Each MOC uses Dataview queries to auto-populate:

```markdown
---
type: moc
domain: [<domain-name>]
---
# <Domain Name>

## Active Projects
```dataview
TABLE status, ai-context
FROM "20_projects"
WHERE contains(domain, "<domain-name>") AND status = "active"
SORT file.mtime DESC
```

## Knowledge Base
```dataview
TABLE ai-context, source
FROM "40_knowledge"
WHERE contains(domain, "<domain-name>")
SORT file.mtime DESC
LIMIT 20
```
```

### Linking Best Practices

- Wrap key concepts in `[[brackets]]` aggressively — creates graph edges
- Use `[[wikilinks]]` over markdown links for Obsidian graph integration
- Link project notes to their domain MOC: `[[moc-creative-studio]]`
